# Memory Share

Cross-IDE memory sharing system for Cursor, Claude Code, and GitHub Copilot via MCP.

> 📖 **New User?** See [USER_GUIDE.md](USER_GUIDE.md) for step-by-step installation and usage instructions.

## Overview

Memory Share enables seamless context handoff between different AI IDEs by maintaining a shared memory backend. When you switch from Claude Code to Cursor, the new session automatically loads the full context from previous sessions. Perfect for developers who frequently switch between IDEs due to token limits or preference for specific tools.

## Features

### Core Capabilities

- **5 MCP Tools**: `memory_sync`, `memory_add_decision`, `memory_manage_task`, `memory_query`, `memory_update_context`
- **2 MCP Prompts**: `sync_memory` (smart sync workflow), `compress_memory` (memory compression)
- **4 MCP Resources**: `memory://briefing`, `memory://tasks`, `memory://decisions`, `memory://status`
- **Smart Sync**: Auto-detects sync direction (pull/push) based on version comparison
- **Intelligent Filtering**: Automatically filters out irrelevant sessions (personal questions, unrelated topics)
- **Token Budget Control**: Automatically trims briefing content to fit token limits

### Memory Management

- **Three-tier Memory Lifecycle**: Hot (48h) → Warm (30d) → Cold (archived)
- **Checkpoint-based Sync**: Incremental pull/push like git, using version numbers
- **Session-based Isolation**: Events are filtered by session, not IDE (allows same-IDE cross-session sharing)
- **Automatic Compaction**: Old events are compressed into digests and archived

### Reliability

- **NFS-safe Locking**: Compatible with network filesystems (uses `mkdir` locks)
- **Atomic Writes**: All file operations use tmp → fsync → rename pattern
- **Consistency Checks**: Automatic repair of version mismatches on startup
- **Idempotency**: Prevents duplicate writes via idempotency keys

### Integration

- **Git Integration**: Automatic commit logging via post-commit hooks
- **Project Scanning**: Auto-generates initial context on `init` (analyzes README, Git log, tech stack)
- **Auto-configuration**: `init` command automatically creates MCP configs for all IDEs

## Quick Start

### 1. Initialize in Your Project

```bash
cd /path/to/your/project
memory-share init
```

This command will:
- Create `.memory/` directory structure
- Scan your project (README, Git log, tech stack)
- Install IDE rule files (`.cursor/rules/memory-share.mdc`, `CLAUDE.md`, `.github/copilot-instructions.md`)
- **Automatically generate MCP configs** for all IDEs:
  - `.cursor/mcp.json` (Cursor)
  - `.mcp.json` (Claude Code)
  - `.vscode/mcp.json` (GitHub Copilot)
- Install Git post-commit hook (if in a Git repo)

### 2. Use in Your IDE

Once initialized, the MCP server is automatically configured. In any IDE session:

- **At session start**: The AI will automatically read `memory://briefing` to load full context
- **During session**: Say "sync memory" / "同步记忆" to trigger smart sync workflow
- **After milestones**: The AI will proactively save progress using `memory_sync(summary="...")`

### 3. CLI Commands

```bash
# Check memory health
memory-share status

# View recent events
memory-share log

# Search memories
memory-share search <keyword>

# List all tasks
memory-share tasks

# View project context
memory-share context

# Compact old events
memory-share compact

# Check system health
memory-share doctor
```

## MCP Tools

### `memory_sync`

Smart bidirectional memory sync. Auto-detects direction based on version comparison.

- **Without summary**: Pulls new events from other IDEs
- **With summary**: Pushes your session progress (LLM-generated digest)
- **With `skip_if_irrelevant=true`**: Skips pushing if session contains no project-relevant information

### `memory_add_decision`

Record an architecture or design decision with reasoning and alternatives.

### `memory_manage_task`

Create, update, complete, or cancel tasks that persist across IDE sessions.

### `memory_query`

Search shared memories by keyword, tags, or source IDE.

### `memory_update_context`

Update project context (current focus, tech stack, description, etc.).

## MCP Prompts

### `sync_memory`

Guides the LLM through the smart sync workflow:
1. Pull new events from other IDEs
2. Evaluate session relevance
3. Summarize and push (or skip if irrelevant)

### `compress_memory`

Guides the LLM through memory compression workflow, filtering out irrelevant events.

## MCP Resources

- **`memory://briefing`**: Full context briefing (active tasks, decisions, recent events, project context)
- **`memory://tasks`**: All tracked tasks with their current status
- **`memory://decisions`**: All recorded architecture and design decisions
- **`memory://status`**: Current sync status (version, sessions, health info)

## Architecture

### Storage Layer

- **File-based**: JSON/JSONL files in `.memory/` directory
- **NFS-safe**: Uses `mkdir` locks instead of `fcntl` for network filesystem compatibility
- **Atomic writes**: All updates use tmp → fsync → rename pattern
- **Version tracking**: Monotonic version counter in `state.json`

### Sync Engine

- **Checkpoint-based**: Each session tracks `last_sync_version`
- **Incremental pull**: Only fetches events with `version > last_sync_version`
- **Session isolation**: Filters by `session_id` (not IDE), allowing same-IDE cross-session sharing
- **CAS operations**: Compare-and-swap for concurrent write safety

### Compaction Engine

- **Hot memory**: Events from last 48 hours (always included in briefing)
- **Warm memory**: Session digests (compressed summaries of old events)
- **Cold memory**: Archived events and completed tasks
- **Automatic archiving**: `compact` command moves old events to `archive/` directory

### Consistency & Reliability

- **Startup consistency check**: Automatically repairs version mismatches
- **Idempotency keys**: Prevents duplicate writes
- **Stale lock recovery**: Automatically cleans up locks older than 30 seconds
- **Event-first writes**: Events are appended before state update (allows recovery on crash)

## File Structure

```
.memory/
├── state.json              # Global sync state (version, sessions)
├── events.jsonl            # Hot memory events (append-only)
├── digests.jsonl           # Warm memory digests (compressed summaries)
├── tasks.json              # All tracked tasks
├── decisions.json           # All recorded decisions
├── context.json             # Project context (single source of truth)
├── context.md              # Dynamically generated from context.json
├── config.json             # Configuration (token budget, TTLs, etc.)
├── .lock/                  # NFS-safe lock directory
├── archive/                # Cold memory (archived events/tasks)
└── backup/                 # Schema migration backups
```

## Usage Examples

### Smart Sync Workflow

When you say "sync memory" in any IDE:

1. **Pull**: The AI calls `memory_sync()` without summary to pull new events
2. **Evaluate**: The AI evaluates if the session contains project-relevant information
3. **Push or Skip**:
   - If relevant: `memory_sync(summary="Implemented X, fixed Y...")`
   - If irrelevant: `memory_sync(skip_if_irrelevant=true)`

### Recording Decisions

```python
# In any IDE session
memory_add_decision(
    title="Use FastMCP for MCP server",
    decision="We use FastMCP 2.0 framework instead of raw MCP SDK",
    reasoning="FastMCP provides cleaner decorator-based API and supports prompts/resources",
    alternatives="Raw MCP SDK was considered but requires more boilerplate",
    tags=["architecture", "mcp"],
    related_files=["src/memory_share/server.py"]
)
```

### Managing Tasks

```python
# Create task
memory_manage_task(
    action="create",
    title="Implement token budget trimming",
    description="Add automatic trimming of briefing content to fit token limits",
    status="in_progress",
    priority="high"
)

# Update task
memory_manage_task(
    action="update",
    task_id="abc123",
    status="completed"
)
```

## Troubleshooting

### Check System Health

```bash
memory-share doctor
```

This checks:
- `.memory/` directory exists
- Data consistency
- Stale locks
- Git hooks installation

### Manual Sync

```bash
# Pull only
memory-share sync --pull

# Push only
memory-share sync --push --summary "Manual sync"

# Both
memory-share sync --both --summary "Manual sync"
```

### View Memory Status

```bash
memory-share status
```

Shows:
- Hot memory stats (events, tasks, decisions)
- Warm memory stats (digests)
- File sizes
- Active sessions
- Warnings (e.g., large events.jsonl)
